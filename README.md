## ✨ 구현 내용

### 1. `@Transactional` 문제 해결

#### 문제
할 일 저장 API 호출 시 데이터가 저장되지 않고 예외가 발생했습니다.  
서비스 클래스에 `@Transactional(readOnly = true)`가 적용되어 있어 저장 메서드까지 읽기 전용 트랜잭션으로 동작하고 있었습니다.

#### 적용 내용
저장이 필요한 메서드에 별도로 `@Transactional`을 선언해 쓰기 가능한 트랜잭션으로 재정의했습니다.

#### 결과
할 일 저장 API가 정상 동작하도록 수정했습니다.  
이를 통해 **조회 전용 트랜잭션과 쓰기 트랜잭션의 차이**를 이해할 수 있었습니다.

---

### 2. JWT에 nickname 추가

#### 문제
기존 User 정보에는 nickname이 없었고, 프론트엔드에서 JWT를 통해 닉네임을 사용할 수 없는 상태였습니다.

#### 적용 내용
- User 엔티티에 `nickname` 컬럼 추가
- 관련 DTO 수정
- JWT Claim에 `nickname` 추가
- JWT 필터 및 사용자 정보 전달 구조에도 nickname 반영

#### 결과
프론트엔드에서 JWT를 통해 사용자 닉네임을 꺼내 사용할 수 있도록 개선했습니다.  
단순히 Claim만 추가하는 것이 아니라 **인증 흐름 전체에 정보가 반영되어야 한다**는 점을 배웠습니다.

---

### 3. JPQL 조건 검색 기능 개선

#### 문제
할 일 검색 시 `weather` 조건 검색과 수정일 기간 검색이 불가능했습니다.  
또한 각 조건은 있을 수도 있고 없을 수도 있어 유연한 검색 처리가 필요했습니다.

#### 적용 내용
JPQL을 활용해 `weather`, `시작일`, `종료일` 조건을 선택적으로 적용할 수 있도록 조회 로직을 수정했습니다.

#### 결과
다양한 검색 조건을 조합해 조회할 수 있도록 개선했습니다.  
이를 통해 **조건이 고정되지 않은 검색 기능을 설계하는 방법**을 익힐 수 있었습니다.

---

### 4. 컨트롤러 테스트 수정

#### 문제
`todo` 단건 조회 시 예외가 발생하는 테스트가 실패하고 있었습니다.  
실제 예외 응답 형식과 테스트 검증 코드가 맞지 않는 상태였습니다.

#### 적용 내용
예외 발생 시 반환되는 응답 구조에 맞게 테스트 코드를 수정했습니다.

#### 결과
실패하던 테스트가 정상적으로 통과하도록 수정했습니다.  
테스트는 내가 예상한 값이 아니라 **실제 응답 형식에 맞게 작성해야 한다**는 점을 다시 확인할 수 있었습니다.

---

### 5. AOP 동작 수정

#### 문제
AOP가 개발 의도와 다르게 동작하고 있었고,  
`changeUserRole()` 메서드 실행 전에 실행되어야 하는 요구사항을 만족하지 못하고 있었습니다.

#### 적용 내용
- 포인트컷 대상을 올바르게 수정
- 실행 시점을 `Before`에 맞게 변경

#### 결과
AOP가 원하는 메서드에서, 원하는 시점에 실행되도록 수정했습니다.  
이를 통해 **AOP는 적용 대상과 실행 시점이 모두 중요하다**는 점을 배웠습니다.

---

### 6. JPA Cascade 적용

#### 문제
할 일을 생성해도 생성한 유저가 담당자로 자동 등록되지 않는 상태였습니다.

#### 적용 내용
Todo와 Manager의 연관관계를 활용하고, Cascade 설정을 적용해 할 일 저장 시 담당자 정보도 함께 저장되도록 수정했습니다.

#### 결과
할 일 생성과 동시에 생성 유저가 담당자로 자동 등록되도록 구현했습니다.  
이를 통해 **연관 엔티티를 함께 저장하는 Cascade의 역할**을 이해할 수 있었습니다.

---

### 7. N+1 문제 해결

#### 문제
댓글 조회 API 호출 시 댓글 작성자 정보를 조회하는 과정에서 N+1 문제가 발생했습니다.

#### 적용 내용
Fetch Join을 사용해 댓글과 사용자 정보를 한 번의 조회로 가져오도록 수정했습니다.

#### 결과
불필요한 추가 쿼리 발생을 줄여 성능 문제를 개선했습니다.  
이를 통해 **일반 Join과 Fetch Join의 차이**를 실제로 이해할 수 있었습니다.

---

### 8. QueryDSL 적용

#### 문제
기존 `findByIdWithUser` 메서드는 JPQL 기반이었고, 유지보수성과 타입 안정성 측면에서 개선이 필요했습니다.

#### 적용 내용
- QueryDSL 설정 추가
- 기존 JPQL 조회 로직을 QueryDSL로 변경
- Fetch Join 적용으로 연관 엔티티를 함께 조회

#### 결과
더 안전하고 읽기 쉬운 조회 로직으로 개선했습니다.  
문자열 기반 쿼리보다 **QueryDSL이 유지보수에 더 유리하다**는 점을 배웠습니다.

---

### 9. Spring Security 적용

#### 문제
기존에는 Filter와 ArgumentResolver를 기반으로 인증/인가를 처리하고 있어 구조가 분산되어 있었습니다.

#### 적용 내용
- 기존 인증 처리 방식을 Spring Security 구조로 변경
- JWT 기반 인증 방식은 유지
- 인증 정보를 SecurityContext에서 관리하도록 수정
- 권한 처리도 Spring Security 방식으로 통합

#### 결과
인증과 인가를 더 일관된 구조로 관리할 수 있게 되었습니다.  
이를 통해 **Spring Security가 왜 표준적으로 많이 사용되는지** 이해할 수 있었습니다.

## 📚 이번 과제를 통해 배운 점

이번 과제를 진행하면서 기능 구현 자체보다,  
**문제가 왜 발생했는지 이해하고 적절한 기술로 해결하는 과정**이 중요하다는 것을 느꼈습니다.

특히 아래 내용을 깊게 학습할 수 있었습니다.

- `@Transactional` 옵션에 따라 저장/조회 동작이 달라진다.
- JWT Claim 추가는 토큰 생성만이 아니라 인증 흐름 전체 수정이 필요하다.
- JPA 연관관계와 Lazy Loading을 이해해야 N+1 문제를 해결할 수 있다.
- QueryDSL과 Spring Security는 코드의 안정성과 유지보수성을 높이는 데 도움이 된다.
